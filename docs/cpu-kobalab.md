# kobalab CPU 解析ドキュメント

## この CPU の位置づけ

`kobalab CPU` は、MOSAIC 独自の Easy / Normal / SOPHIA とは別系統の CPU です。  
現行コードでは `src/game/kobalab.ts` に実装されており、`UpperHand` 由来ロジックを MOSAIC に適応した移植系 CPU として扱われています。

- 通常難易度の延長ではなく、比較研究対象
- MOSAIC 側の評価型 CPU と違い、探索ベースの色が強い
- 盤面表現を一度フラットなテンプレートに変換してから考える

研究価値があるのは、「MOSAIC 側が直接作った判断」と違う思考法を持ち込めるからです。

## 一言でいうと

kobalab CPU は、「盤面全体を上に伸びる依存構造として見て、価値の高い位置から順に読みを入れ、ミニマックス系の探索で手を選ぶ CPU」です。

## 基本的な手選びの流れ

現行実装の流れは次のとおりです。

1. MOSAIC の `board[level][row][col]` を、kobalab 側が使うフラットな `status[]` に変換する
2. 各位置について、そこがどの上位位置に影響するかを表すテンプレートを使う
3. `KobalabCpu.selectMove()` が優先順に合法手を調べる
4. 各手について `evaluate()` を呼び、探索結果の評価値を比較する
5. 最も評価値の高い手を採用する

Easy / Normal / SOPHIA のように「全合法手をスコア式で一括採点」するのではなく、  
kobalab は「探索しながら候補を順に試す」感触が強いです。

## 候補手の集め方

### フラットな状態への変換

`createKobalabGameFromState` では、各盤面位置をテンプレート順に1本の配列へ並べます。

- `-1`: 今すぐ合法手として置ける位置
- `null`: まだ支持が足りず置けない位置
- `0`: 中立
- `1`: blue
- `2`: yellow

この表現に変えることで、kobalab CPU は「どこが合法か」だけでなく、「その位置が上位構造にどう影響するか」を一貫して扱えます。

### 候補の優先順位

`makePriorityOfPosition` は、各位置が将来どれだけ多くの上位マスへ影響するかを `getWeight` で数え、その重い順に位置を並べます。

人間向けに言えば、

- 上に波及しやすい場所
- 将来の構造に関わりやすい場所

から先に注目する並びです。

MOSAIC 版では、同点の位置をランダムに並べ替えず、決定的順序に固定しています。  
これは比較研究しやすくするための調整です。

## 何を評価しているか

kobalab CPU の評価は、MOSAIC 側の「加点・減点一覧」とはかなり違います。  
鍵になるのは `getValue()` と `evaluate()` です。

### 1. 各位置の“期待値”を下から上へ伝播させる

`getValue()` は、各位置に 0.0 から 1.0 の値を割り当てます。

- Player 1 の駒: `1.0`
- Player 2 の駒: `0.0`
- まだ打てる空点 `-1` や中立 `0`: `0.5`
- まだ未確定な上位位置 `null`: 子4点の値から計算

上位位置は子4点の値から、  
「4つとも有利」「3つ有利」「2つずつなら半々」  
という発想で値が作られます。

つまり kobalab は、個々の手の良し悪しを直接採点するより、

- この盤全体が Player 1 にどれだけ寄っているか
- まだ未確定の上位構造が、どちらへ傾きそうか

を全体期待値として見ています。

### 2. 盤面全体の合計価値を見る

`getValue()` は各位置の値を合計し、その盤面全体の評価値 `rv` を返します。  
これは局所利得より「盤面全体の支配期待」に近い指標です。

人間向けに言えば、

- ただ今すぐ取れる得点
- ただ今すぐ防げる危険

ではなく、

- この盤全体が将来どちらのものになりやすいか

を見ています。

### 3. 探索 `evaluate()`

`evaluate()` はミニマックス系の再帰探索です。  
現在手番が探索対象プレイヤーと同じなら最大化し、違うなら最小化します。  
さらに `a`, `b` を使って枝刈りもしています。

これにより kobalab CPU は、

- 自分が打ったあと
- 相手がどう返し
- さらにその先でどうなるか

を深さ制限つきで比較します。

MOSAIC 側 CPU との最大差はここで、kobalab は「相手返しの最大値」を評価項目のひとつとして足すのではなく、探索そのもので扱います。

## あえて見ていないこと

kobalab CPU は探索型ですが、MOSAIC 向けの現行実装として見ていないものもあります。

- MOSAIC 独自の解説しやすい評価分解は持たない
- 「緊急防御」「自爆予約」などの専用安全項目は別立てしていない
- 終盤だけ重みを切り替える明示的フェーズ分けはない
- 打ち手理由を人間が追いやすい形に分解して返さない

つまり kobalab は、

- 全体期待値と探索には強い
- ただし「なぜその手か」を項目別に説明しにくい

タイプです。

## 局面ごとの振る舞い

### 序盤

`depth()` は、まだ Player 2 の石が盤上にない間は `0` を返します。  
このときは探索せず、優先順位配列の先頭から合法手を探して即決します。

つまり序盤の最初期は、

- 上位構造に影響しやすい位置
- 優先順位の高い位置

を機械的に取りに行く傾向があります。

これは MOSAIC 側 CPU のような「中央寄り」「軽い形評価」とは別種の序盤感です。

### 中盤

Player 2 の石が現れると探索が始まります。  
`level` が 3 の場合、探索深さは

- 合法手数が多い局面では浅く
- 合法手数が減るほど深く

なります。  
式は `Math.max(Math.floor(50 / moves.length), 3)` なので、局面が進んで選択肢が減るほど深読みになります。

このため中盤では、

- 候補が多いときは大局の優先順位を保ちつつ浅く読む
- 候補が絞れてくると探索色が強くなる

という性格が出ます。

### 終盤

終盤は合法手数が減るため、探索深さが自然に上がりやすくなります。  
明示的な「終盤補正」はありませんが、結果として読む量が増え、終盤らしく締まっていきます。

MOSAIC 側 CPU は終盤で重みを切り替えますが、kobalab は探索深さの変化で終盤性が出る、と見ると分かりやすいです。

## ランダム性と揺らぎ

kobalab CPU は、現行 MOSAIC 実装ではかなり決定的です。

- 同点優先順位は固定
- `selectMove()` は評価値最大の最初の手を選ぶ
- Easy / Normal / SOPHIA のようなノイズ加算はない

そのため、同じ局面なら同じ手を返しやすいです。  
比較検証用としては扱いやすく、プレイ感としては少し機械的に見えることがあります。

## 人間向けに言い換えると

kobalab CPU は、人間でいえば次のような思考です。

- 盤面を局所の得失ではなく、上へつながる全体構造として見る
- 将来に効きやすい位置から順に注目する
- その先の応酬を浅めから中程度に読み、悪い返しを避ける
- 説明可能性より、構造的な期待値と探索を優先する

棋風としては、「局所の気分ではなく、盤全体の支配期待で考える探索派」に近いです。

## なぜそういう手を選びやすいのか

kobalab CPU が独特に見えるのは、MOSAIC 側 CPU と見ている単位が違うからです。

- MOSAIC 側 CPUは「この一手で何が増えるか」を分解して見る
- kobalab CPUは「盤全体の期待値がどちらへ傾くか」を見る

そのため人間には、

- 目先の自動積み上がりを取りに行かない
- 一見地味な位置を早めに確保する
- 盤面全体の流れを優先しているように見える

ことがあります。

## 強み

- 探索型なので、相手返しを構造的に扱える
- 位置の将来影響度を優先するため、盤全体の価値を見失いにくい
- 局面が狭まるほど自然に深く読める
- 同一局面で再現性が高く、研究比較に向く

## 弱み

- 目先の危険を人間向けの安全指標として分解していない
- MOSAIC 固有の勝ち筋や受け渡し概念に最適化されているわけではない
- 序盤最初期は探索せず優先順位即決なので、やや定型的に見える
- なぜその手かを人間が追うには、評価値の意味を翻訳する必要がある

## 研究・比較対象としての価値

kobalab CPU の価値は、強いか弱いかだけではありません。

- MOSAIC 独自評価型と異なる思考枠組みを持っている
- 全体期待値と探索を中心にした別文化の CPU として比較できる
- 将来、評価型 CPU と探索型 CPU、さらに ML 系 CPU を比べるときの橋渡しになる

SOPHIA が「手作り評価の多面的比較」なら、kobalab は「構造期待値 + 探索」の比較対象です。

## 今後の改善余地

- 現行の探索値を、人間向けに読める補助ログへ変換する
- MOSAIC 特有の終盤勝ち切りや受け渡し危険を、補助評価として足す
- 序盤最初期の即決部分に少しだけ変化を持たせるか再検討する
- 研究モード向けに、評価値や探索深さを表示できるようにする
