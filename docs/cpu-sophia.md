# SOPHIA CPU 解析ドキュメント

## この CPU の位置づけ

`SOPHIA` は、現行 Web 版における最上位の評価型 CPU です。  
コード上の内部識別子は後方互換のため `hard` のままですが、実装は Easy / Normal とは別段階です。

- Easy / Normal は共通の軽量評価式を重み違いで使う
- SOPHIA は `chooseHardMove` と `scoreHardMove` を中心にした専用ロジックを持つ
- kobalab CPU とは別系統で、MOSAIC 側が独自に育てている高度評価型と見てよい

用途としては、

- 現行 Web 版で最も厳しい通常 CPU
- 人間強者との比較や改善議論の土台
- 将来の ML 系 CPU と比べる際の、手作り評価の基準線

に当たる存在です。

## 一言でいうと

SOPHIA は、「全合法手を一度シミュレートし、目先の得、形の伸び、相手最善返し、自分の自爆予約、終盤の勝ち切りやすさをまとめて比べる CPU」です。

## 基本的な手選びの流れ

SOPHIA の処理は、Easy / Normal より一段複雑です。

1. `getLegalMoves` で全合法手を列挙する
2. `resolveHardEndgamePhase` で局面を `normal / endgame / late_endgame` に分ける
3. 各手に対し `scoreHardMove` を実行する
4. 終盤では `endgameAdjustment` を使って候補をゲートで絞る
5. 残った候補の中から、小さな揺らぎ付きで選ぶ

重要なのは、SOPHIA はただ点数を付けるだけでなく、終盤だけ候補集合そのものを絞ることです。  
「どれを選ぶか」だけでなく、「終盤にどの手はもう候補に残さないか」を考えています。

## 候補手の集め方

入口では全合法手を見ます。  
危険手を事前除外するのではなく、全件に対してかなり細かい評価をかけます。

- 全合法手列挙
- 各手を `placeManualPiece` ベースでシミュレート
- 相手の返しも合法手全件で見る
- 終盤だけ `endgameAdjustment` の高い手に絞る

この意味で SOPHIA は、「候補手を集める CPU」ではなく「候補手を精密にふるいにかける CPU」です。

## 評価していること

SOPHIA の本体は `scoreHardMove` です。  
ここで見ているものを人間向けに訳すと、主に次の7系統です。

### 1. 即時利益 `evaluateHardImmediateValue`

まず、その手を打ってすぐ起きる利益を数えます。

- 自分の自動積み上がり数
- 自分の自動積み上がりがどの高さまで伸びたか
- 相手色の自動積み上がりが混ざったか
- 連鎖の深さ
- その一手で自分が何個駒を消費できたか

ここでの「自分が何個駒を消費できたか」が重要です。  
MOSAIC は残り駒を 0 にした側が勝つため、SOPHIA にとって自動配置を含む駒消費はそのまま勝利への前進です。

ただし、序盤からこの即時利益だけで走るわけではありません。  
後述のとおり、局面段階によってこの項目の重みは落とされます。

### 2. 局所的な形の伸び `evaluateHardLocalPatternDelta`

SOPHIA は着手点周辺の 2x2 を見て、形の変化をかなり細かく分解します。

- 自分の 1個形が 2個形に伸びたか
- 自分の 2個形が 3個形に伸びたか
- 複数の支援形が同時に増えたか
- 相手の 3個形を壊したか
- 相手の 2個形の伸びを抑えたか

Easy / Normal の局所形評価よりも、人間の感覚に近いです。  
単に「同色が何個あるか」ではなく、「その一手で何段階進んだか」を見ています。

### 3. 緊急防御 `evaluateHardUrgentThreatBlock`

これは SOPHIA らしい項目です。  
着手近傍の 2x2 に対して、

- 相手が 3個同色 + 1空きで、あと1手で即完成する形
- 相手が 2対1優勢 + 1空きで、次に3化しやすい形

を壊せたかを見ています。

つまり SOPHIA は、「ただ危険を減らす」ではなく、「今すぐ放置するとまずい窓を閉じたか」を別枠で評価しています。  
これにより、守りの必要な局面で分かりやすく硬くなります。

### 4. 将来の発火力 `evaluateHardPatternPotential` と `countPatternPotential`

SOPHIA は、今の即時利益だけでなく、「この局面に次の連鎖の芽がどれだけ残っているか」も見ます。

`countPatternPotential` は、各 2x2 について上に置けるかを確認したうえで、

- 3個同色の強い候補
- 2個同色 + 中立入りの候補
- それらが近接して密集しているか

を数えます。

人間向けに言えば、

- すでに半分できている攻め筋
- 中立を足場にして育ちそうな筋
- 候補が固まっていて次の連鎖の網になりそうか

を見ています。

これがあるため SOPHIA は、「今すぐ連鎖しないが、あとで効く形」をそれなりに評価できます。

### 5. 相手への受け渡しリスク `evaluateHardOpponentReply`

ここが SOPHIA の中核です。  
自分が候補手を打ったあとの局面で、相手の全合法返しを列挙し、相手側にとって最も価値の高い返しを探します。

相手返しの評価軸は次の合計です。

- 相手の即時利益
- 相手の局所的な形の伸び
- 相手の将来発火力
- その返しで自分の将来発火力をどれだけ抑えられるか

さらに、その返しが相手の即勝ちなら大きな追加ペナルティを入れます。

これを人間向けに言うと、SOPHIA は

- 「この手のあと、相手が一番嫌な返しをしたらどうなるか」
- 「その返しは単に得なだけか、それとも即死級か」

をかなり真面目に見ています。

Easy / Normal との決定的な差はここです。

### 6. 自分の自爆予約 `evaluateHardSelfCompletionRiskPenalty`

SOPHIA は、自分の手が「あと1マス空いている自分3個形」に触れてしまうかも見ています。  
ここで見ているのは、自分にとって良い形そのものではなく、

- 今は自分3個 + 1空きで完成待ち
- そこへ自分で置いてしまうと、その予約枠を自分の手置きで埋めてしまう

というケースです。

人間向けに言えば、「自動発火で取れたはずの上積みを、自分の手置きで雑に消していないか」を見ています。  
これにより SOPHIA は、目先の着手で将来の自動完成を潰す手を嫌います。

### 7. 連鎖の逆噴射 `evaluateHardChainBackfirePenalty`

MOSAIC では、自分の手がきっかけでも、連鎖結果として相手色の自動積み上がりが走ることがあります。  
SOPHIA はこれをかなり警戒します。

- 相手の自動積み上がり数が自分より多いか
- 相手の連鎖深さが自分より深いか
- 相手の上段到達量が自分より大きいか
- とくに相手残り駒が少ない終盤で、その逆噴射が致命傷にならないか

人間の感覚でいうと、「起爆したのは自分でも、取り分が相手に偏る連鎖」を嫌っています。

### 8. 終盤価値 `evaluateHardEndgameAdjustment`

終盤では、SOPHIA は単に連鎖を褒めるだけでなく、

- 何個消費できたか
- 残り駒をどこまで減らせたか

を別枠で評価します。

MOSAIC の終盤では「大きい形」より「確実に減らすこと」の方が大切になるため、これはかなり実戦的です。

## 局面ごとの振る舞い

SOPHIA は局面段階を明示的に分けています。

### 段階判定 `resolveHardEndgamePhase`

- `late_endgame`
  - どちらかの残り駒がかなり少ない、または合計残り駒がかなり少ない
- `endgame`
  - その一歩手前
- `normal`
  - それ以外

### その段階で何が変わるか

`resolveHardPhaseMultipliers` により、

- 即時利益
- 形の伸び
- 相手返しリスク

の重みが終盤ほど下がります。

一見すると不思議ですが、これは「終盤は細かな総合評価より、勝ち切りやすさで候補を絞る」ためです。  
終盤では先に `endgameAdjustment` でゲートをかけ、その範囲内で総合点を見る流れになっています。

### 序盤

- 相手返しリスクや形の伸びをしっかり比較する
- すぐの得だけでは飛びつきにくい
- まだ消費勝負より局面品質を重視する

### 中盤

- 各評価軸の競合が最も激しい
- 利得と危険の天秤が強く働く
- SOPHIA らしい「総合判断」が一番見える

### 終盤

- 勝ち切りやすい手を残すために候補ゲートを入れる
- ランダム性もかなり減らす
- 夢のある形より、消費と残り駒差を重視する

## ランダム性と揺らぎ

SOPHIA にもランダム性はありますが、かなり弱いです。

### 基本の揺らぎ

- `noiseRange = 0.8`
- `nearBestMargin = 5`
- `poolLimit = 2`

つまり通常局面では、「ほぼ最善近辺の2手くらいで少し揺れる」程度です。

### 終盤の揺らぎ

`late_endgame` ではさらに候補を絞り、ノイズも小さくします。  
終盤で毎回大きくぶれないのはこのためです。

### それでも完全固定にしない理由

- 評価が近い同質手を毎回同じにしすぎないため
- 対局の再現性と変化のバランスを取るため
- 実戦相手として少し自然さを残すため

## あえて見ていないこと

SOPHIA は現行最上位ですが、万能ではありません。  
見ていないものもはっきりあります。

- 本格的な多手先探索はしていない
- 相手返しは1手先までで、その先の読み合いは評価に折り込まない
- 評価は足し算中心で、局面の意味を学習で獲得しているわけではない
- 盤全体の長期計画より、今作れる形と今渡す危険に寄りやすい

そのため、非常に複雑な2手先3手先の罠では、人間強者の方が深く読める余地があります。

## 人間向けに言い換えると

SOPHIA は人間でいえば、次のような思考です。

- まず全部の合法手を見る
- 目先の連鎖や消費量を数える
- その一手で周囲の形がどう伸びるかを見る
- 相手にとって最悪の返しを1手だけ真面目に想像する
- 自分で将来の好形を潰していないかも点検する
- 終盤なら勝ち切りやすい手だけを候補に残す

「1手先の返しをかなり重視する評価派の強者」に近い棋風です。

## なぜ不自然な手を選ぶことがあるのか

SOPHIA の手が人間に不自然に見えるのは、たいてい評価軸の競合が原因です。

### 1. 目先の大連鎖を見送る

人間が「ここは派手に行ける」と感じても、SOPHIA は

- 相手最善返しの打点
- 連鎖逆噴射
- 自爆予約

を重く見て、地味な受け手を選ぶことがあります。

### 2. 形を壊すような手を打つ

局所形の見た目が悪くても、

- 緊急防御が大きい
- 相手3個形を壊せる
- 相手返しリスクを大きく減らせる

なら高評価になります。

### 3. 終盤で急に実利一辺倒になる

終盤は候補ゲートと `endgameAdjustment` が強く効くため、  
中盤までなら選びそうな育成手より、露骨に駒を減らす手へ寄ります。

これは不自然というより、設計として明示的にそうしています。

## 評価軸どうしの競合

SOPHIA を理解するうえで重要なのは、各項目がしばしばぶつかることです。

- 即時利益は高いが、相手返しリスクも高い
- 局所形は伸びるが、自爆予約も生じる
- 守りとしては優秀だが、終盤消費は進まない
- 将来発火力は高いが、今の危険窓を閉じられない

SOPHIA はこれらを総合点で折り合わせるため、人間の単一評価では説明しづらい手が出ます。  
逆に言えば、その複合判断こそが現行 Web 版での SOPHIA の価値です。

## 強み

- Easy / Normal より明確に受け渡しに強い
- 局所形の育成と緊急防御の両立ができる
- 相手最善返しを1手先でかなり真面目に見る
- 終盤で勝ち切り寄りの判断に切り替わる
- 自分が起こした連鎖が相手有利になるケースを警戒できる

## 弱み

- 読みの深さはまだ1手返し中心で、多手先の罠には限界がある
- 評価式のバランス次第で、特定局面に偏った選択をする可能性がある
- 終盤ゲートが強いため、人間目線では少し早めに夢を切ることがある
- 「本当に重要な1項目」が足し算の中で埋もれるケースは理論上残る

## 今後の改善余地

- 相手返しのさらに先まで、限定的にでも読む仕組みを加える
- 評価項目の競合関係をログ可視化し、PDCA を回しやすくする
- 終盤ゲートの閾値や multipliers を実戦データで再調整する
- 手作り評価を維持しつつ、ML 系の補助評価と比較できる形に整理する
